# ガイドライン：講座メンテナンス・品質管理

**対象**: 講座メンテナンス・品質管理部門  
**最終更新**: 2026-05-18  
**ステータス**: 育成中

---

## 1. 更新判定プロセス

講座の更新が必要かどうかを判定するプロセスです。

### ステップ1: 字幕内容の分析

**参照場所**: `projects/reference/subtitle-library/[講座名]/subtitles/ja.vtt`

字幕ファイルから以下を確認：

- **リリース日**: 講座フォルダ名から確認
- **技術情報**: 言語バージョン、ツールバージョン、APIバージョン等
- **情報の鮮度**: 「2024年時点では」などの時間参照
- **推奨ツール**: OSやソフトウェア依存部分
- **業界トレンド**: ベストプラクティス、標準化の動き

### ステップ2: 技術トレンド調査

更新判定に必要な調査：

| 項目 | 確認方法 | 更新の判断基準 |
|------|--------|--------|
| **言語バージョン** | 公式ドキュメント確認 | メジャーバージョン変更で要検討 |
| **ライブラリ更新** | GitHub releases/ドキュメント | 破壊的変更があれば更新必要 |
| **仕様変更** | サービス公式ニュース | 講座内容と不整合で更新必要 |
| **ベストプラクティス** | 業界ニュース・ブログ | 推奨方法が大きく変われば更新 |

### ステップ3: 受講生フィードバック確認

コミュニティ運営部門から以下の情報を収集：

- 「このセクションは古い」という指摘
- 「動きません」「エラーが出ます」という報告
- 「代わりにXXの方法が一般的になった」といった質問

### ステップ4: 更新の必要性を判定

```
更新判定フロー

字幕から見た情報の古さ
  ↓
【古い】 → トレンド調査
  ↓ 【最新情報と大きく異なる】
  ↓
受講生フィードバック確認
  ↓ 【問題報告あり】
  ↓
【更新が必要】
```

**判定結果**:
- **即時更新推奨**: 技術が古く、受講生から問題報告がある
- **3-6ヶ月内に更新**: 技術は古いが、受講生は困っていない（ただし早期更新推奨）
- **継続監視**: 現在のところ問題なし（3ヶ月ごとに再確認）
- **更新不要**: 技術的な変化なし、受講生満足度が高い

## 2. 更新優先度の判定基準

複数の講座の更新が必要な場合、優先度を決定します：

### 優先度「高」（即実施）
- 受講生から「動きません」という報告が多数
- セキュリティアップデートが必要
- 廃止予定の技術を使用している

### 優先度「中」（1-3ヶ月以内）
- 新しいバージョンに破壊的変更がある
- ベストプラクティスが大きく変わった
- 受講生からの「古い」という指摘が増加

### 優先度「低」（3-6ヶ月内）
- マイナーアップデート程度
- 説明の精度向上程度
- 受講生からの指摘がない

---

## 3. 更新内容の判定

何をどの程度更新するかを判定します：

### 軽微な更新（1-2日）
- 説明の言い回しを修正
- スクリーンショット数枚を更新
- バージョン番号の更新

### 中程度の更新（1-2週間）
- セクション1-2個の内容更新
- 古いツール → 新しいツールへの置き換え
- 補足資料の追加・削除

### 大型リニューアル（3-4週間以上）
- 複数セクションの全面改編
- カリキュラム構成の変更
- 学習目標の変更

---

## 4. 診断レポート作成

更新判定の結果を報告します：

**レポートに含める内容**:

1. **鮮度度評価**
   - 公開からの経過時間
   - 関連技術の変化度合い

2. **具体的な問題点**
   - 「XのバージョンYは既に廃止されている」
   - 「ベストプラクティスが変わった」

3. **推奨アクション**
   - 何を更新すべきか
   - 更新の優先度
   - 想定される作業時間

4. **実施予定時期**
   - 「高優先度：即実施」
   - 「中優先度：2026-06月内」
   - 「低優先度：2026-07月以降」

## 5. よくある判定ケース

### ケース1: 技術バージョンアップ（Python 3.9 → 3.12）
```
判定: 更新必要（優先度：中）
理由: 構文変更あり、新機能活用で説明改善可能
実施予定: 2026-06月内
```

### ケース2: APIエンドポイント変更
```
判定: 更新必要（優先度：高）
理由: 受講生が「動きません」と報告、コード実行不可
実施予定: 即実施
```

### ケース3: ベストプラクティスの進化（マイナーな変化）
```
判定: 継続監視（優先度：低）
理由: 説明内容は正確だが、より良い方法が登場
実施予定: 3-6ヶ月内検討
```
