## 1. 文体・表現ルール

### 基本原則

**敬語**: 「です・ます」で統一
```
✅ 「このアプローチが有効です」
❌ 「このアプローチが有効だ」
```

**能動態を優先**
```
✅ 「営業チームが顧客を訪問した」
❌ 「顧客が営業チームに訪問された」
```

**短い文**: 1文30字以内
```
✅ 「売上は50%増加しました。主な要因はキャンペーン施策です。」
❌ 「売上が50%増加した理由は主にキャンペーン施策の実施によるものであり、これにより顧客獲得が加速したためである。」
```

**専門用語は定義**
```
✅ 「CAC（顧客獲得単価）が低下しました」
❌ 「CACが低下しました」（定義なし）
```

### 避けるべき表現

| 避ける | 理由 | 改善案 |
|-------|------|--------|
| 「〜かもしれない」 | 不確実 | 「〜と考えられます」 |
| 「一般的に〜」 | 根拠不明 | 「調査結果では〜」 |
| 「皆さん ご存じ」 | 前提を強要 | 「〜の背景として」 |
| 「など」（具体例なし） | 曖昧 | 具体例を列挙 |

## 2. 主要ドキュメントのテンプレート

### 戦略提案書

```markdown
## [タイトル]

### 背景・課題
[なぜこの提案が必要か]

### 提案内容
- [施策1]: [概要と理由]
- [施策2]: [概要と理由]

### 期待される効果
- KPI1: 現状 → 目標値
- KPI2: 現状 → 目標値

### リスク・制約
- [リスク]: [対策]
- [制約]: [対応]

### 実行体制と期間
[誰が、いつまでに実施するか]

### 決定・承認ポイント
[何をいつまでに決めるべきか]
```

### 分析レポート

```markdown
## [分析タイトル]

### 結論
[最初に3行で結論を述べる]

### 分析データ
[データと数値表示]

### 考察・分析
[なぜそうなったのか、何が見えるのか]

### 推奨アクション
- [対策1]: [実施方法]
- [対策2]: [実施方法]

### 参考データ・出典
[データソース明記]
```

### 業務マニュアル

```markdown
## [業務名]

### 目的
[この業務の目的]

### 前提条件
[実行前に確認すべきこと]

### 手順
1. [ステップ1] → [確認事項]
2. [ステップ2] → [確認事項]
3. ...

### よくある質問と対応
Q: [質問]
A: [回答]

### トラブルシューティング
[問題が起きた場合の対応]
```

## 3. メール・報告文体

### 報告メールの構成

```
【件名】: 
[重要度]_[内容の要約]
例: 【報告】キャンペーンA ROI達成

本文:
[1行で全体像]

[詳細内容（3-5行）]

[推奨アクション］

[添付資料］
```

### よい件名の例
```
✅ 「【決定待ち】新規事業投資判定(月末期限)」
❌ 「重要な報告です」

✅ 「【完了】Q2予算編成」
❌ 「予算について」
```

### メール本文のコツ
- 最初の1行で「誰が」「何をしたのか」を述べる
- 詳細は3行程度（長い場合は別紙）
- 「~についてご確認ください」などで次のアクション明示

## 4. ビジュアル・グラフ表示ルール

### グラフの選択

| データ | グラフ種 | 例 |
|--------|--------|-----|
| **時系列** | 折れ線グラフ | 売上推移 |
| **比較** | 棒グラフ | 部門別売上 |
| **割合** | 円グラフ | 顧客セグメント比率 |
| **相関** | 散布図 | 広告費 vs 売上 |

### グラフ作成時のチェック

- [ ] タイトルが分かりやすいか
- [ ] 軸ラベルが明記されているか（単位含む）
- [ ] 凡例がある（色のみで区別しない）
- [ ] データソースが明記されているか
- [ ] 過度に複雑でないか（1グラフ1メッセージ）

## 5. 数値・統計表示

### 有意性の表示

```
✅ 「増加率20%（前年比）」
❌ 「20%増加」（何との比較か不明）

✅ 「NPS 52点（目標50点）」
❌ 「NPS達成」（数値不明）
```

### 小数点以下の表示

- **金額**: 円単位（小数点なし）
- **パーセンテージ**: 小数点第1位まで（例：15.3%）
- **KPI**: 計画時に精度を定義（後から変更しない）

### 大きな数値の表示

```
✅ 「1,500万円」「52%」
❌ 「15000000」「0.52」
```

## 6. 報告・会議での話し方

### プレゼンテーションの構成

1. **冒頭**: 結論を先に述べる（結論ファースト）
2. **詳細**: 3つのポイントに絞る（多すぎない）
3. **クロージング**: 「以上です。質問はありますか」

### よくあるプレゼン失敗と対策

| 失敗 | 対策 |
|------|------|
| **結論が分かりにくい** | 最初に30秒で結論を述べる |
| **データが多すぎる** | 重要データ3つに絞る |
| **時間超過** | 事前にリハーサル、時間配分決定 |
| **聞き手を見ない** | 最初と最後は聞き手を見る |

## 7. 質問・確認のコミュニケーション

### 効果的な質問

```
❌ 「これでいいですか」（Yes/No質問）
✅ 「この部分をどう改善するのがいいと思いますか」（開かれた質問）

❌ 「なぜできなかったのか」（責める）
✅ 「このステップで課題がありましたね。どうなりました」（事実確認）
```

### フィードバックの与え方

```
1. 肯定（良かった点を述べる）
2. 建設的指摘（改善点を述べる）
3. 期待（次に期待することを述べる）
```

**例**: 「このカリキュラムは学習目標が明確でいいですね。加えて、各ユニットの所要時間を記載すると、さらに実用的になります。次回はそこを含めていただけると幸いです。」

## 8. ドキュメント管理

### ファイル名ルール

```
[日付]_[内容]_[版].xlsx
例: 20260513_Q2決算_v1.xlsx

[担当者]_[内容]_[完了日].md
例: ID_カリキュラム設計書_20260513.md
```

### バージョン管理
- **v1, v2, ...**: 重大な変更時に増分
- **最終版**: 確定後は「final」または日付付き

### 保存場所
- 共有フォルダ: プロジェクトごとの階層
- バックアップ: 月1回の全体バックアップ
- アーカイブ: 1年以上前の資料は別フォルダ

## 9. よくあるコミュニケーション失敗と対策

| 失敗 | 対策 |
|------|------|
| **報告忘れ** | スケジュールに「報告」を組み込む |
| **曖昧な報告** | 数値・具体例を含める |
| **一方的な指示** | 相手の意見を聞く時間を作る |
| **長過ぎるメール** | 最初の段落で全体像、詳細は別紙 |
